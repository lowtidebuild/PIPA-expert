## Problem Statement

ChatGPT Custom GPT, Gemini Gem 등 기존 방식으로 PIPA(개인정보보호법) 전문 AI를 만들어봤지만, flat document upload → 의미 검색 방식은 법령의 계층 구조, 교차참조, 소스 등급을 무시해서 실무에서 신뢰하기 어렵다. 인하우스 변호사가 실제로 쓸 수 있는 수준의 PIPA 전문 Agent를 만들려면, 법령 데이터를 구조화된 형태로 전처리하고 계층적으로 관리하는 Agent 전용 RAG 시스템이 필요하다.

## What Makes This Cool

**데이터 구조가 곧 지능이다.** 기존 GPT/Gem은 "더 똑똑한 검색"을 추구하지만, 이 시스템은 "더 똑똑한 데이터"를 추구한다. PIPA 조문을 계층적으로 구조화하고, 시행령/고시와의 관계를 명시적으로 매핑하면, 검색 엔진이 단순해도 결과가 정확하다. Agent가 "제15조 제1항의 시행령 위임 조문"을 물어볼 때, flat text에서는 환각이 발생하지만 structured data에서는 정확한 경로를 따라간다.

**범용 법령 RAG 아키텍처.** PIPA로 시작하되, 어떤 법령이든 같은 구조로 추가 가능한 범용 설계. GDPR, AI기본법, 상법 등 확장 가능.

## Constraints

- 오픈소스/개인 학습 프로젝트
- law.go.kr Open API 활용 (크롤링 최소화)
- Claude Code Agent가 소비하는 RAG (사람 UI가 아닌 Agent 도구)
- Phase 1은 PIPA 단독, Phase 2에서 다른 법령 추가
- 기존 PIPA+GDPR Control Tower 디자인의 Source Hierarchy (Grade A-D) 체계 재활용

## Premises

1. Custom GPT/Gem의 한계는 데이터 구조의 부재에서 온다 (검색 엔진의 한계가 아님)
2. law.go.kr Open API를 통해 PIPA 법령 데이터를 구조화된 형태로 수집할 수 있다
3. 조문 단위 청킹 + 메타데이터 + 교차참조가 flat text 임베딩보다 검색 품질이 높다
4. 범용 법령 RAG 구조를 설계하면 PIPA 이후 다른 법령도 같은 구조로 추가 가능하다
5. Agent가 소비하는 RAG이므로 사람이 읽기 편한 것보다 Agent가 정확히 참조할 수 있는 구조가 우선이다

## Approaches Considered

### Approach A: File-System RAG (Agent-Native)
- Agent의 내장 도구(Grep/Glob/Read)로 직접 탐색하는 구조화된 파일 시스템
- 조문별 .md + YAML frontmatter, 폴더 계층 = 법령 계층
- Effort: S | Risk: Low
- Pros: 추가 인프라 없이 즉시 동작, 환각 최소, 파일 추가/수정 간단
- Cons: 의미 검색 없음 (Grep 키워드만), 수천 파일 시 검색 느려짐

### Approach B: Hybrid File + Vector RAG (Ideal Architecture)
- A와 동일한 .md 구조 + 벡터 임베딩 검색 레이어
- Agent가 의미 검색 → 매칭된 .md 파일 직접 읽기
- Effort: M | Risk: Med
- Pros: 의미 검색 + 파일 구조 유지, 검색 품질 최상
- Cons: 벡터 DB 인프라 필요, 임베딩 파이프라인 구축/유지

### Approach C: Ontology Graph RAG (Creative/Lateral)
- 법령을 그래프로 모델링 (노드=조문, 엣지=관계)
- Effort: L | Risk: High
- Pros: 관계 쿼리에 최강, 개정 이력 추적
- Cons: 그래프 DB 필요, 초기 구축 비용 높음

## Recommended Approach

**Approach B: Hybrid File + Vector RAG** — A의 파일 구조를 기반으로 벡터 검색을 추가. A부터 완성한 뒤 벡터 레이어를 얹는 점진적 구축.

### 항 분할 규칙

기본적으로 조문(article) 전체를 하나의 .md 파일로 관리. 단, 항(paragraph) 본문이 500자(원문 raw text 기준, 마크다운 문법 제외)를 초과하는 경우 항별로 분할:
- 파일명: `art15-p1.md`, `art15-p2.md` (p = paragraph 번호)
- 각 분할 파일은 동일 frontmatter 스키마 사용, `paragraph` 필드에 항 번호 기재
- `article-index.json`에서 분할된 조문은 항별로 별도 엔트리: `"id": "pipa-art15-p1"`, `"path": "...art15-p1.md"`
- 실무상 대부분의 PIPA 조문은 500자 이내이므로 분할 빈도는 낮을 것으로 예상

### 법령 개정 버전 관리

법령이 개정될 때:
1. 현재 조문 파일을 `archive/{law}/{개정일}/`에 복사 (개정 전 스냅샷)
2. `library/` 내 조문 파일을 최신 버전으로 갱신
3. frontmatter의 `effective_date`, `last_amended` 업데이트
4. `_meta.json`에 개정 이력 추가

Agent는 기본적으로 최신 버전(`library/`)을 참조. "개정 전 조문"을 물어보면 `archive/`에서 해당 시점 스냅샷을 Read.

### 데이터 갱신 정책

| 소스 유형 | 갱신 주기 | 방법 |
|----------|----------|------|
| 법령 원문 (Grade A) | 법령 개정 시 | `check-freshness.py`로 API 대비 변경 감지 → 수동 확인 후 갱신 |
| 가이드라인 (Grade A) | 30일 | PIPC 공지 수동 모니터링 |
| 처분례 (Grade B) | 30일 | PIPC 공개 결정 수동 확인 |
| 판례 (Grade B) | 수시 | 대법원 신규 판결 수동 확인 |

Phase 1에서는 수동 갱신. Phase 2에서 `check-freshness.py`로 반자동화 (API 응답의 last-modified와 `retrieved_at` 비교).

### 벡터 임베딩 전략 (Phase 2)

1. **청킹 단위:** 조문(article) 단위 기본. 항(paragraph)이 긴 경우 항 단위로 분할.
2. **임베딩 모델:** multilingual-e5-large 또는 ko-sroberta-multitask (한국어 법률 도메인 성능 비교 필요)
3. **Summary-Augmented Chunking:** 각 청크에 해당 조문의 요약 + 상위 장/절 컨텍스트를 prepend
4. **메타데이터 필터링:** 벡터 검색 시 source_grade, law, chapter로 필터링 가능
5. **저장소:** ChromaDB (경량, Python 네이티브, 로컬 실행)

### Agent 사용 패턴

Agent가 PIPA 관련 질문을 받았을 때:

**Phase 1 검색 흐름 (파일 기반):**
```
1. article-index.json 읽기 → keywords 배열에서 질문 키워드와 부분 문자열(substring) 매칭
   (예: "수집하는" → "수집" 매칭. 한국어 형태소 분석은 Phase 2에서 도입)
2. 매칭된 조문의 path로 .md 파일 목록 확보 (상위 5개)
3. 키워드 매칭이 부족하면 (2개 미만) Grep으로 library/ 전체에서 본문 검색
4. 매칭된 .md 파일을 Read로 직접 읽기
```

**Phase 2 검색 흐름 (벡터 + 파일):**
```
1. 질문 → 벡터 DB (ChromaDB) → 의미 유사도 top-5 조문 반환
2. 반환된 .md 파일을 Read로 직접 읽기
```

- "과거에 ChatGPT Custom GPT, Gemini Gem으로 유사한 시스템을 만들어본적 있는데, 그것보다 훨씬 deep하게 가려고" — 얕은 버전을 직접 경험하고 그 한계를 체감한 뒤에 더 깊은 버전을 만드려는 것. 이건 실험 → 학습 → 개선 사이클을 자연스럽게 돌리는 사람의 패턴이다.
- "범용 법령 RAG"을 선택한 것 — PIPA 하나에 집중할 수도 있었지만, 처음부터 확장 가능한 구조를 요구한 것은 아키텍처 감각. 한 번 만들면 다시 만들기 싫다는 실용적 판단.
- "내가 데이터를 모아올테니까 너가 좋은 방법을 생각해줄래?" — 역할 분담을 명확히 한 것. 자기가 잘하는 것(도메인 지식, 데이터 수집)과 AI가 잘하는 것(구조 설계)을 구분하는 능력.
- "AGENT 전용 RAG" — 사람이 읽을 UI가 아니라 Agent가 소비할 데이터라는 관점 전환. 이건 AI 도구를 단순히 쓰는 게 아니라 AI의 작동 원리를 이해하고 최적화하는 레벨.
